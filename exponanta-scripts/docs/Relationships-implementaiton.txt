Exactly. Two modes:

Mode 1 — Mixed doctypes (current RelationshipPanel)

Rows are different doctypes — Customer, User, Task etc
Only system fields make sense as columns — related_title, type, _state badge, FSM buttons
Schema in_list_view fields are meaningless — each row has different schema
Mode 2 — Single doctype (scoped grid)

All rows are same doctype — e.g. all Customer
Schema in_list_view fields apply — show name, status, priority etc
Behaves like MainGrid but scoped to relationships of parent doc
Config drives the mode:

javascript
// Mode 1 — mixed, array config
"Task": {
  "User": ["Assignee", "Reviewer"],
  "Customer": ["Customer"],
}

// Mode 2 — single doctype slot, object config
"Task": {
  "Customer": {
    "Customer": { reqd: 1, max: 1, grid: true } // grid:true → Mode 2
  }
}
Or field-level:

json
{ "fieldname": "relationships", "fieldtype": "Relationship Panel", "options": "Customer" }
field.options set → Mode 2. field.options empty → Mode 1.

field.options is already in your schema field structure — simplest trigger for mode switch, no config changes needed.

What they all converge on:

For your situation, the config should live in the schema field definition:

json
{
  "fieldname": "relationships",
  "fieldtype": "Relationship Panel",
  "options": "Customer",
  "grid": {
    "mode": "single",
    "columns": ["related_title", "status", "type"],
    "actions": ["submit", "cancel"],
    "reqd": 1,
    "max": 1
  }
}
Key insight from OSS grids: columns and datasource are always separate concerns. Your _getListFields already handles columns. Your run_doc.child handles datasource. The grid config just wires them together.

What you don't need that OSS grids have:

Virtual scrolling — your datasets are small
Cell editing — you have MainForm for that
Column resizing/reordering — overkill for now
Simplest viable approach: extend the schema field with a grid object. RelationshipPanel reads it and switches mode. Everything else — FSM buttons, fetch, key management — stays unchanged.

Claude responded: This is the classic per-user vs per-schema tension.
This is the classic per-user vs per-schema tension. Best practice from OSS grids:

Three levels, each overriding the previous:

Schema default — in_list_view fields, defined once, applies to all
Per-doctype user preference — stored in localStorage or a UserPreference doctype, keyed by doctype + view, applies to all documents of that type for that user
Per-document override — stored on the document itself, rarely needed
For your situation:

Schema default is already done via in_list_view. That's level 1 — covers 90% of cases.

Level 2 — per-doctype user preference — is the sweet spot:

javascript
// key: "grid_columns:Task:relationships"
// value: ["subject", "status", "priority", "owner"]
CW._getGridColumns = function(doctype, fieldname) {
  const key = `grid_columns:${doctype}:${fieldname}`;
  const saved = localStorage.getItem(key);
  return saved ? JSON.parse(saved) : CW._getListFields(run_doc);
};
Level 3 — per-document — only makes sense for something like a personal dashboard or saved view. Don't implement unless asked.

Recommendation: implement level 2 only. Schema defines defaults, user can customize per doctype, stored in localStorage. Simple, universal, non-breaking. Per-document is powerful but creates maintenance burden — every document becomes a unique snowflake.
