jsfunction computeDiff(before, after) {
  const diff = {};
  for (const k of new Set([...Object.keys(before), ...Object.keys(after)])) {
    if (JSON.stringify(before[k]) !== JSON.stringify(after[k])) {
      diff[k] = { before: before[k], after: after[k] };
    }
  }
  return diff;
}

Why strict at write time:
1. Immutability — if you calculate diff later from two stored snapshots, both snapshots must be stored. That's double storage. If either is tampered with, the diff is wrong. At write time you have ground truth — before is the DB state, after is what was committed.
2. Auditor expectation — SOC 2 auditors want to see "what changed in this operation" as a single atomic record. Not "here are two snapshots, please diff them yourself."
3. Forensics — if a record is later deleted or corrupted, you still have the field-level diff in the audit log. Deferred calculation loses this.
4. Performance — calculating diff at write time is cheap (one pass over changed fields). Calculating diff later requires fetching two records and comparing — expensive at audit query time.
5. NIST 800-53 AU-10 — non-repudiation requires the audit record to be complete and self-contained at the time of the event. Deferred diff breaks non-repudiation.
Practical concern: the _before snapshot on run_doc is cheap — JSON.parse(JSON.stringify(doc)) before merge. The diff calculation after completion is one computeDiff call. Total overhead is negligible.
Recommendation: snapshot before _mergeInput, diff after adapter completes, persist both diff and final state in one audit record. Strict, complete, self-contained.
