Waterfall of Ownership

1. Schema — declares intent
javascriptCW.Schema['HomePage'] = {
  fields: [
    { fieldname: 'tasks', fieldtype: 'Table',    options: 'Task',
      run_args: { query: { where: { parent: '{{doc.name}}', status: 'Open' } } } },
    { fieldname: 'reviews', fieldtype: 'ChildRun',
      run_args: { target_doctype: 'Review', view: 'block' } },
    { fieldname: 'body',  fieldtype: 'BlockNote' },
    { fieldname: 'files', fieldtype: 'Filepicker' },
  ]
}
Schema owns: field definition, fetch config (run_args), type declaration. Nothing else.

2. CW.fieldTypes registry — maps fieldtype to lifecycle config
javascriptCW.fieldTypes = {
  Table:      { expand: 'parent',   component: 'UniversalGrid'  },
  ChildRun:   { expand: 'run_args', component: 'ChildRunComponent' },
  BlockNote:  { expand: null,       component: 'BlockNoteField' },
  Filepicker: { expand: null,       component: 'Filepicker'     },
  Data:       { expand: null,       component: 'DataField'      },
  Link:       { expand: 'lazy',     component: 'LinkField'      },
}
Registry owns: which pre-fetch strategy, which component. Nothing else.

3. _expand — pre-fetch phase
javascript// reads CW.fieldTypes[field.fieldtype].expand
// 'parent'   → query: { where: { parent: docName } }
// 'run_args' → spread field.run_args, CW._resolveQuery
// 'lazy'     → skip, component fetches on demand
// null       → skip, no pre-fetch needed
_expand owns: spawning child runs before render, one per expandable field. Nothing else.

4. FieldRenderer — dispatch phase
javascriptconst ftDef = CW.fieldTypes[field.fieldtype]
const Comp  = globalThis[ftDef.component]
return ce(Comp, { field, run_doc, readOnly })
FieldRenderer owns: reading registry, mounting correct component. Nothing else.

5. Component — UI + interaction lifecycle
javascript// reads pre-fetched child run
const child = CW._getChildRun(run_doc, field.fieldname)

// owns from here:
// - local state
// - user interactions
// - further child runs (upload, lazy fetch, signal)
// - re-render decisions
Component owns: everything after mount — state, interactions, child run firing, re-render. Nothing else.

The complete waterfall
Schema          →  what fields exist + fetch config
fieldTypes      →  which strategy + which component
_expand         →  pre-fetch child runs in parallel
_render         →  mount MainForm → FieldRenderer per field
FieldRenderer   →  dispatch to component by fieldtype
Component       →  own UI + interaction + further child runs
Each layer owns exactly one concern. Nothing crosses boundaries. Data flows down via run_doc. Three reasons.

1. Explicit run_args is genuinely new
Table, Link, Block all have implicit fetch logic hardcoded in _expand. ChildRun makes the fetch declaration explicit and data-driven:
javascript// Table — _expand hardcodes the query
query: { where: { parent: docName } }

// ChildRun — query is declared on the field
run_args: { query: { where: { event: '{{doc.event_id}}' } } }
Any query, any filter, any ordering — without writing a new fieldtype. That's real flexibility Table and Block don't have.

2. It's the escape hatch for non-standard fetches
Table always fetches by parent. Link always fetches by name. ChildRun fetches by anything:
javascript// can't express with Table or Block
{ fieldtype: 'ChildRun', run_args: { 
    target_doctype: 'Event',
    query: { where: { organizer: '{{doc.name}}', status: 'Active' }, order: 'date ASC', take: 3 }
}}

3. run_args spread into run_doc.child() directly
No intermediate mapping. _expand just spreads field.run_args. That's the cleanest possible API — the field IS the function call.

Where it's unnecessary
For standard parent-child relationships — Table and Block defaults — ChildRun is more verbose without adding value. { fieldtype: 'Table', options: 'Task' } is cleaner than { fieldtype: 'ChildRun', run_args: { target_doctype: 'Task', query: { where: { parent: '{{doc.name}}' } }, view: 'list' } }.

Verdict
Keep ChildRun as the power user fieldtype — for non-standard queries, custom views, computed filters. Keep Table, Link, Block as ergonomic shorthands for the common cases.
